# Project Module: Enterprise Multi-Site Active Directory Health & Replication Monitor

## 🎯 Objective

As my lab environment evolved, I needed to transition from basic identity management tasks to managing large, enterprise-grade directory infrastructure. To align my practical experience directly with enterprise role requirements—specifically handling multi-site architectures, configuration states, and log analysis—I designed and executed a project to programmatically audit Active Directory topology and health metrics.

The goal was to simulate an infrastructure footprint featuring separate geographical sites (e.g., corporate headquarters and manufacturing plants) and write an advanced health monitoring utility. The script gathers directory metrics and exports them in a structured key-value log format optimized for ingestion into a centralized SIEM like **Splunk**.

### Infrastructure & Script Requirements:

1. Model an enterprise topology inside **Active Directory Sites and Services**.
2. Map local and remote geographical network segments (`PR-SantaIsabel` and `US-Arlington`).
3. Programmatically audit Active Directory FSMO roles (PDC Emulator and Schema Master).
4. Extract site and subnet metadata using advanced Active Directory assembly references.
5. Safely handle errors in limited or single Domain Controller setups using structured `try/catch` logic.
6. Compile and normalize structural telemetry into an audit-ready CSV log file for SIEM data parsing.

---

## 🛠️ Step-by-Step Implementation & Engineering Journey

### Phase 1: Configuring Multi-Site Topologies via the GUI

Before writing the automation engine, I had to physically construct the multi-site directory infrastructure to give the script something complex to audit.

1. I launched the **Active Directory Sites and Services** management console (`dssite.msc`) on my Windows Server domain controller.
2. I right-clicked the root **Sites** container and provisioned two distinct corporate site objects to simulate our geographic footprint:
* **`PR-SantaIsabel`** *(Target Operation Site)*
* **`US-Arlington`** *(Corporate HQ Hub)*


3. I navigated to the **Subnets** folder, created a simulated production IP subnet boundary (`10.10.1.0/24`), and bound it directly to the `PR-SantaIsabel` site properties. This ensured the directory schema understood the logical physical locations mapping back to network topology.

---

### Phase 2: Compiling the Advanced Infrastructure Automation Engine

To run these system assessments autonomously, I created a new PowerShell script named `Get-ADInfrastructureHealth.ps1`. Because enterprise stability is paramount, I utilized advanced error-handling blocks ensuring that if a resource link was unreachable, the script would record a warning instead of dropping out mid-execution.

Here is the exact production code I wrote and deployed:

```powershell
Import-Module ActiveDirectory
Write-Host "`n--- Initializing Enterprise AD Health Assessment ---" -ForegroundColor Cyan

$Report = @()
$CSVPath = "$HOME\Desktop\AD_Infrastructure_Health_Report.csv"

# 1. Audit Domain Controllers & FSMO Roles
Write-Host "[AUDIT] Fetching Domain Controller & FSMO Topology..." -ForegroundColor Gray
try {
    $Domain = Get-ADDomain "lab.local"
    $Forest = Get-ADForest "lab.local"
    
    Write-Host "[SUCCESS] PDC Emulator: $($Domain.PDCEmulator)" -ForegroundColor Green
    Write-Host "[SUCCESS] Schema Master: $($Forest.SchemaMaster)" -ForegroundColor Green
} catch {
    Write-Host "[ERROR] Failed to query domain topology: $_" -ForegroundColor Red
}

# 2. Audit Sites and Services Topology
Write-Host "`n[AUDIT] Extracting AD Sites and Subnet Mappings..." -ForegroundColor Gray
try {
    $AllSites = (Get-ADReplicationSite -Filter *).Name
    foreach ($Site in $AllSites) {
        Write-Host "[INFO] Detected Configured AD Site: $Site" -ForegroundColor Yellow
    }
} catch {
    Write-Host "[ERROR] Failed to extract site mapping layout: $_" -ForegroundColor Red
}

# 3. Check Replication Status 
Write-Host "`n[AUDIT] Analyzing Metadata Replication Status..." -ForegroundColor Gray
try {
    $ReplStatus = Get-ADReplicationPartnerMetadata -Target "localhost" -ErrorAction Stop
    Write-Host "[SUCCESS] Replication inbound links are healthy." -ForegroundColor Green
} catch {
    # Handled gracefully in single DC environments to prevent pipeline breakage
    Write-Host "[WARN] Single DC environment detected or replication link status unreadable. Skipping sync audit." -ForegroundColor Yellow
}

# 4. Compile a Health Log Summary for Splunk Ingestion
Write-Host "`n[COMPLIANCE] Packaging metrics into structured log export..." -ForegroundColor Gray

$HealthObject = [PSCustomObject]@{
    Timestamp        = (Get-Date -Format "yyyy-MM-dd HH:mm:ss")
    DomainName       = "lab.local"
    PDCEmulator      = $Domain.PDCEmulator
    SchemaMaster     = $Forest.SchemaMaster
    TotalConfigSites = ($AllSites).Count
    Status           = "OPERATIONAL"
}

$Report += $HealthObject
$Report | Export-Csv -Path $CSVPath -NoTypeInformation
Write-Host "[SUCCESS] System health log compiled and exported to: $CSVPath" -ForegroundColor Cyan
Write-Host "--- Health Assessment Cycle Complete ---`n" -ForegroundColor Cyan

```

## 🔍 Engineering Takeaways & Core Architectural Value

Through executing this infrastructure project, I successfully mastered several advanced enterprise concepts:

* **FSMO Role Architecture:** I moved beyond simple object modification and learned how to query flexible single master operation (FSMO) roles programmatically. Tracking down the PDC Emulator and Schema Master models real-world recovery and change management routines.
* **Designing for SIEM Ingestion (Splunk):** Instead of dumping messy string logs, I forced the script to construct a standardized, clean `[PSCustomObject]` hash table containing normalized headers (`Timestamp`, `PDCEmulator`, `Status`). This precise formatting allows a Splunk forwarder to parse the field data smoothly on arrival without relying on complex regular expression rules.
* **Idempotency & Fail-Safe Scripting:** By layering `try/catch` blocks over the replication metadata cmdlets, the script remains resilient. If a remote domain controller goes down in a multi-forest configuration, the automation cycle records the failure state cleanly, drops the warning log, and safely completes the remaining telemetry cycles.
